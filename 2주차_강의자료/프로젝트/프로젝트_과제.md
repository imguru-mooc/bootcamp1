# 캡스톤 미션 (하나의 문제)

> **미션** — 노트북 카메라가 손을 보고 **펴진 손가락 개수(0~5)** 를 인식한다. 그 개수에 따라 〔Pi〕의 **DC 모터**(전진·후진·정지), **스테핑 모터**(정·역방향 회전), **LCD**(현재 동작 표시)를 제어한다. 전 과정을 **런치 파일로 한 번에 기동**해 시연하고, 제스처 매핑과 한계를 **발표**한다.

이 과제는 여러 개의 작은 문제가 아니라 **하나의 통합 시스템**입니다. Day 1~4에서 각자 따로 동작시킨 부품들이 **한 줄의 흐름**으로 이어집니다.

```
[카메라] → [손 인식: 손가락 개수] → [제스처 매핑] → [DC 모터 / 스테퍼 / LCD]
```

핵심은 **인식(perception)** 과 **정책(policy)** 의 분리입니다. 인식 노드는 "손가락이 몇 개인가"만 책임지고, 코디네이터 노드가 "그 개수로 무엇을 할 것인가"를 결정합니다. 같은 손가락 개수라도 매핑을 바꾸면 로봇의 행동이 바뀝니다 — 이 매핑 설계가 캡스톤의 핵심입니다.

# 성공 기준

다음을 **하나의 실행(런치 1회)** 으로 모두 보이면 미션 성공입니다.

- 카메라에 손을 비추면 **펴진 손가락 개수가 정확히 인식**된다(LCD 또는 디버그 영상으로 확인).
- 손가락 개수에 따라 **DC 모터가 전진·후진·정지**한다.
- 특정 제스처에서 **스테핑 모터가 정방향/역방향으로 회전**한다(제스처가 바뀔 때 한 번씩).
- 현재 동작 이름이 **LCD에 표시**된다(예: `FORWARD`, `STEP CW`, `STOP`).
- 손을 화면에서 치우면(개수 0) 로봇이 **안전하게 정지**한다.

# 시스템 구성 — 데이터 흐름

실행 위치를 색으로 구분합니다. 〔노트북〕은 인식·판단(두뇌), 〔Pi〕는 구동·표시(몸).

```text
                              〔노트북〕
  camera ─▶ webcam_pub ─▶ /image_raw ─▶ gesture_recognizer ─▶ /finger_count ─┐
                                              │                               │
                                              └─▶ /gesture/image(디버그)       ▼
                                                                  gesture_coordinator (신규)
                                                                   │      │       │
                                              /cmd_vel ◀───────────┘      │       └──────────▶ /lcd_text
                                                                  /stepper_steps
  ─────────────────────────────────────────────────────────────────────────────────────────────────
                              〔Pi〕
   dc_motor_node ◀─ /cmd_vel        stepper_node ◀─ /stepper_steps        lcd_display ◀─ /lcd_text
   (L293D, 전진/후진/정지)          (ULN2003/28BYJ-48, 정·역방향)         (I2C LCD1602, 동작 표시)
```

**토픽 계약(반드시 일치시킬 이름)**

| 토픽 | 타입 | 발행 | 구독 | 의미 |
|---|---|---|---|---|
| `/image_raw` | `sensor_msgs/Image` | webcam_pub〔노트북〕 | gesture_recognizer | 원본 영상 |
| `/finger_count` | `std_msgs/Int32` | gesture_recognizer〔신규〕 | gesture_coordinator | 펴진 손가락 개수 0~5 |
| `/gesture/image` | `sensor_msgs/Image` | gesture_recognizer | (rqt_image_view) | 랜드마크 디버그 영상 |
| `/cmd_vel` | `geometry_msgs/Twist` | gesture_coordinator〔신규〕 | dc_motor_node〔Pi〕 | DC 모터 속도·방향 |
| `/stepper_steps` | `std_msgs/Int32` | gesture_coordinator〔신규〕 | stepper_node〔Pi〕 | 스텝 수(+정방향/−역방향) |
| `/lcd_text` | `std_msgs/String` | gesture_coordinator〔신규〕 | lcd_display〔Pi〕 | 현재 동작 표시 문자열 |

> 💡 이 표가 **시스템의 계약서**입니다. 노드를 누가 만들었든 토픽 이름·타입만 맞으면 연결됩니다. 통합이 안 될 때는 코드보다 먼저 이 표와 `ros2 topic list`를 대조하세요.

# 제스처 매핑 설계 (직접 정의)

손가락 개수(0~5)를 어떤 동작에 연결할지 **개인이 직접 표를 채워** 결정합니다. 아래는 **권장 기본안**이며, 그대로 써도 되고 개인 아이디어로 바꿔도 됩니다(바꾸면 발표에서 이유를 설명).

| 손가락 | DC 모터(`/cmd_vel`) | 스테퍼(`/stepper_steps`) | LCD(`/lcd_text`) |
|---|---|---|---|
| 0 (주먹) | 정지 | — | `STOP` |
| 1 (검지) | 전진 | — | `FORWARD` |
| 2 (브이) | 후진 | — | `BACKWARD` |
| 3 | 정지 | 정방향 1바퀴 | `STEP CW` |
| 4 | 정지 | 역방향 1바퀴 | `STEP CCW` |
| 5 (손바닥) | 정지 | — | `READY` |

**두 가지 명령 성격을 구분하세요.**
- **연속 명령(DC 모터)** — 손가락을 펴고 있는 동안 계속 같은 속도. 그래서 매 프레임 `/cmd_vel`을 갱신해 모터 워치독을 유지합니다.
- **엣지 명령(스테퍼·LCD)** — 제스처가 **바뀌는 순간 한 번만** 발행합니다. 매 프레임 발행하면 스테퍼가 끝없이 돌아가니, 코디네이터 골격의 `changed` 분기를 사용합니다.

# 필수 요구사항 명세

하나의 시스템이 만족해야 할 조건입니다(개별 채점 문제가 아니라 **통합 동작의 조건**).

- **R1 · 분산 통신** 〔노트북〕·〔Pi〕가 같은 `ROS_DOMAIN_ID`, `ROS_LOCALHOST_ONLY=0`으로 토픽을 주고받는다(Day 2).
- **R2 · 인식** 카메라 영상에서 손을 찾아 펴진 손가락 개수를 `/finger_count`로 발행한다(MediaPipe, Day 4 기법).
- **R3 · 매핑** `gesture_coordinator`가 손가락 개수를 위 표대로 `/cmd_vel`·`/stepper_steps`·`/lcd_text`로 변환한다.
- **R4 · DC 모터** 통과한 `/cmd_vel`로 `dc_motor_node`가 전진·후진·정지한다(워치독 포함, Day 3).
- **R5 · 스테핑 모터** `/stepper_steps`로 `stepper_node`가 정·역방향 회전한다. **제스처가 바뀔 때만 한 번** 동작한다(Day 3).
- **R6 · LCD** `/lcd_text`로 `lcd_display`가 현재 동작을 표시한다(Day 3).
- **R7 · 디바운스** 손 떨림으로 명령이 튀지 않도록, 같은 제스처가 **N프레임 연속**일 때만 반영한다.
- **R8 · 일괄 기동** 노트북 측 파이프라인을 **런치 파일 하나**로 띄운다(Day 1·2 런치).
- **R9 · 안전 정지** 손이 화면에서 사라지면(개수 0) DC 모터가 **정지**하고, `Ctrl+C` 시에도 워치독으로 멈춘다.

# 구현 로드맵 (한 프로젝트의 4단계)

미션은 하나지만, 한꺼번에 켜면 어디서 막혔는지 모릅니다. **부품 → 인식 → 매핑 → 통합** 순서로 쌓아 올리세요. 각 단계 끝에 체크포인트가 있습니다.

## 1단계 · 부품 재점검 (개별 동작)
Day 3 액추에이터가 각자 살아 있는지 토픽으로 직접 찔러봅니다.

```bash
# 〔Pi〕 세 액추에이터 띄우기
$ export GPIOZERO_PIN_FACTORY=lgpio
$ python3 ~/ros2_labs/dc_motor_node.py
$ python3 ~/ros2_labs/stepper_node.py
$ python3 ~/ros2_labs/lcd_display.py

# 〔노트북〕 토픽으로 직접 명령해 동작 확인
$ ros2 topic pub --once /cmd_vel geometry_msgs/Twist '{linear: {x: 0.5}}'
$ ros2 topic pub --once /stepper_steps std_msgs/Int32 '{data: 512}'
$ ros2 topic pub --once /lcd_text std_msgs/String '{data: "HELLO"}'
```

> ✅ **체크포인트 1** — `/cmd_vel`로 바퀴가 돌고, `/stepper_steps`로 스테퍼가 한 바퀴 돌고, LCD에 글자가 뜬다.

## 2단계 · 손가락 개수 인식
신규 `gesture_recognizer`의 `count_fingers`를 완성해 개수를 발행합니다.

```bash
# 〔노트북〕 (webcam_pub 실행 중)
$ python3 ~/ros2_labs/webcam_pub.py
$ python3 ~/ros2_labs/gesture_recognizer.py
$ ros2 topic echo /finger_count                 # 손가락 0~5 정확히 나오는지
$ ros2 run rqt_image_view rqt_image_view        # /gesture/image 로 랜드마크 확인
```

> ✅ **체크포인트 2** — 손가락을 0~5개 펴면 `/finger_count`가 그대로 따라온다.

## 3단계 · 제스처 → 하드웨어 매핑
신규 `gesture_coordinator`의 TODO를 채워 매핑을 완성합니다. 먼저 토픽 출력만 확인한 뒤 Pi에 연결합니다.

```bash
# 〔노트북〕
$ python3 ~/ros2_labs/gesture_coordinator.py
$ ros2 topic echo /cmd_vel                       # 손가락에 따라 전진/후진/정지
$ ros2 topic echo /stepper_steps                 # 제스처 바뀔 때만 한 번
$ ros2 topic echo /lcd_text                       # 동작 이름
```

> ✅ **체크포인트 3** — 손가락 1=전진, 2=후진, 3·4=스테퍼 1회전, 0=정지가 토픽에 정확히 나오고, 스테퍼 명령이 **매 프레임 쏟아지지 않는다**.

## 4단계 · 통합 기동 + 발표 준비
런치 하나로 노트북 파이프라인을 띄우고, Pi 액추에이터에 연결해 **시연 영상**을 찍습니다.

> ✅ **체크포인트 4** — `ros2 launch ... capstone_gesture.launch.py` 한 줄로 인식·코디네이터가 뜨고, 손동작만으로 DC·스테퍼·LCD가 반응한다.

# 신규 노드 ① — 제스처 인식 골격

손 랜드마크에서 펴진 손가락 개수를 세어 `/finger_count`로 발행합니다. `count_fingers`의 `TODO`를 채우세요(나머지 배관은 완성되어 있습니다).

`~/ros2_labs/gesture_recognizer.py` (노트북)

```python
#!/usr/bin/env python3
# 〔노트북〕 제스처 인식 노드 (골격)
#   입력 : /image_raw (Image)                 ← Day4 webcam_pub
#   출력 : /finger_count (Int32: 0~5)          → gesture_coordinator
#          /gesture/image (Image: 디버그 영상) → rqt_image_view
#   ※ count_fingers()의 TODO를 채워 손가락 개수 판정을 완성하세요.
import cv2
import rclpy
from rclpy.node import Node
from sensor_msgs.msg import Image
from std_msgs.msg import Int32
from cv_bridge import CvBridge
import mediapipe as mp

TIPS = [4, 8, 12, 16, 20]      # 손가락 끝 랜드마크
PIPS = [3, 6, 10, 14, 18]      # 손가락 둘째 관절


class GestureRecognizer(Node):
    def __init__(self):
        super().__init__('gesture_recognizer')
        self.bridge = CvBridge()
        self.hands = mp.solutions.hands.Hands(
            max_num_hands=1, min_detection_confidence=0.6)
        self.draw = mp.solutions.drawing_utils
        self.create_subscription(Image, 'image_raw', self.on_image, 10)
        self.count_pub = self.create_publisher(Int32, 'finger_count', 10)
        self.img_pub = self.create_publisher(Image, 'gesture/image', 10)
        self.get_logger().info('제스처 인식 시작 — /image_raw → /finger_count')

    def count_fingers(self, lm, handedness):
        fingers = 0
        # ─────────────────────────────────────────────
        # TODO 1. 엄지: 손 방향(handedness: 'Right'/'Left')에 따라
        #         lm[4].x 와 lm[3].x 를 비교해 펴짐(+1) 판정.
        # TODO 2. 검지~새끼: 끝(TIPS[1:])이 관절(PIPS[1:])보다
        #         위(y가 더 작음)면 펴짐(+1).  zip(TIPS[1:], PIPS[1:]) 활용.
        # 힌트) Day4 4-6-1 gesture_control 의 count_fingers 참고. 반환 0~5.
        # ─────────────────────────────────────────────
        return fingers

    def on_image(self, msg):
        frame = self.bridge.imgmsg_to_cv2(msg, 'bgr8')
        rgb = cv2.cvtColor(frame, cv2.COLOR_BGR2RGB)
        result = self.hands.process(rgb)
        count = 0
        if result.multi_hand_landmarks:
            hand = result.multi_hand_landmarks[0]
            label = result.multi_handedness[0].classification[0].label
            count = self.count_fingers(hand.landmark, label)
            self.draw.draw_landmarks(
                frame, hand, mp.solutions.hands.HAND_CONNECTIONS)
        self.count_pub.publish(Int32(data=count))   # 손 없으면 0 → 코디네이터가 정지
        cv2.putText(frame, f'fingers: {count}', (10, 30),
                    cv2.FONT_HERSHEY_SIMPLEX, 1.0, (0, 255, 0), 2)
        self.img_pub.publish(self.bridge.cv2_to_imgmsg(frame, 'bgr8'))


def main(args=None):
    rclpy.init(args=args)
    node = GestureRecognizer()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()
```

> 💡 셀카처럼 좌우가 반전되면 엄지 판정의 부등호가 반대가 됩니다. `webcam_pub`에서 `cv2.flip(frame, 1)`을 쓰는지 확인하고, 인식이 반대면 부등호를 뒤집으세요(Day 4 트러블슈팅).

# 신규 노드 ② — 제스처 코디네이터 골격

손가락 개수를 받아 DC 모터·스테퍼·LCD 명령으로 바꿉니다. **디바운스와 엣지 검출 배관은 완성되어 있고**, `TODO 1~3`의 **매핑만** 채우면 됩니다. 이것이 캡스톤의 핵심 코드입니다.

`~/ros2_labs/gesture_coordinator.py` (노트북)

```python
#!/usr/bin/env python3
# 〔노트북〕 제스처 코디네이터 (골격)
#   입력 : /finger_count (Int32: 0~5)          ← gesture_recognizer
#   출력 : /cmd_vel (Twist)        → Day3 dc_motor_node〔Pi〕   (연속 명령)
#          /stepper_steps (Int32)  → Day3 stepper_node〔Pi〕    (엣지 1회)
#          /lcd_text (String)      → Day3 lcd_display〔Pi〕     (바뀔 때)
#   ※ TODO 1~3 의 제스처 → 하드웨어 매핑을 직접 정의하세요.
import rclpy
from rclpy.node import Node
from std_msgs.msg import Int32, String
from geometry_msgs.msg import Twist

# ── 튜닝 파라미터 (실제 로봇에서 보정) ──
DRIVE_SPEED  = 0.5      # m/s, 전진/후진 기본 속도
STEP_ONE_REV = 512      # 28BYJ-48 한 바퀴 ≈ 512 하프스텝
STABLE_N     = 3        # 같은 제스처가 N프레임 연속이어야 인정(떨림 방지)


class GestureCoordinator(Node):
    def __init__(self):
        super().__init__('gesture_coordinator')
        self.cand = -1          # 후보 제스처
        self.cand_n = 0         # 후보 연속 횟수
        self.last_acted = -1    # 마지막으로 동작을 실행한 제스처(엣지 검출용)
        self.create_subscription(Int32, 'finger_count', self.on_count, 10)
        self.cmd_pub = self.create_publisher(Twist, 'cmd_vel', 10)
        self.step_pub = self.create_publisher(Int32, 'stepper_steps', 10)
        self.lcd_pub = self.create_publisher(String, 'lcd_text', 10)
        self.get_logger().info('제스처 코디네이터 시작 — /finger_count 구독')

    def on_count(self, msg):
        count = msg.data
        # ── 디바운스: 같은 값이 STABLE_N회 연속일 때만 인정 ──
        if count == self.cand:
            self.cand_n += 1
        else:
            self.cand, self.cand_n = count, 1
        if self.cand_n < STABLE_N:
            return
        stable = self.cand
        changed = (stable != self.last_acted)   # 제스처가 바뀐 순간인가?

        # ── (A) 연속 명령: DC 모터는 매번 갱신해 워치독을 유지한다 ──
        cmd = Twist()
        # TODO 1. 손가락 개수(stable) → DC 모터 속도 매핑.
        #   예) 0=정지(0.0), 1=전진(+DRIVE_SPEED), 2=후진(-DRIVE_SPEED)
        #       그 외는 정지. cmd.linear.x 에 대입.
        self.cmd_pub.publish(cmd)

        # ── (B) 엣지 명령: 스테퍼·LCD는 '바뀔 때 한 번만' (매 프레임 발행 금지) ──
        if changed:
            # TODO 2. 손가락 개수 → 스테퍼 동작. 엣지에서만 발행.
            #   예) 3=정방향 한 바퀴(+STEP_ONE_REV), 4=역방향(-STEP_ONE_REV)
            #       해당 없으면 발행하지 않는다.

            # TODO 3. 손가락 개수 → LCD 텍스트(제스처 이름).
            #   예) 0:"STOP" 1:"FORWARD" 2:"BACKWARD" 3:"STEP CW" 4:"STEP CCW" 5:"READY"
            text = ''
            self.lcd_pub.publish(String(data=text))

        self.last_acted = stable


def main(args=None):
    rclpy.init(args=args)
    node = GestureCoordinator()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()
```

> ⚠️ 메서드 이름으로 `handle`·`context`·`clock`·`executor`를 쓰지 마세요(rclpy 예약어와 충돌해 조용히 멈춥니다). 위 골격은 `on_count`만 사용합니다.

> 💡 **왜 스테퍼는 `changed`에서만 발행할까요?** 콜백은 카메라 프레임마다 호출됩니다. 매번 `/stepper_steps`를 보내면 스테퍼가 멈추지 않고 계속 돕니다. 제스처가 **바뀐 순간**에만 한 번 보내야 "손가락 3 → 한 바퀴" 동작이 됩니다.

```python
import os
from launch import LaunchDescription
from launch.actions import ExecuteProcess

LABS = os.path.join(os.path.expanduser('~'), 'ros2_labs')


def run(script):
    return ExecuteProcess(cmd=['python3', os.path.join(LABS, script)], output='screen')


def generate_launch_description():
    return LaunchDescription([
        run('webcam_pub.py'),            # 카메라 → /image_raw
        run('gesture_recognizer.py'),    # /image_raw → /finger_count
        run('gesture_coordinator.py'),   # /finger_count → /cmd_vel,/stepper_steps,/lcd_text
```

# 설계 결정·유의사항

- **연속 vs 엣지** — DC 모터는 매 프레임 갱신(워치독 유지), 스테퍼·LCD는 제스처가 **바뀔 때만** 발행. 이 구분을 코드로 지키지 않으면 스테퍼가 멈추지 않습니다.
- **디바운스 필수** — 손가락 인식은 프레임마다 1~2개씩 튑니다. `STABLE_N`으로 같은 값이 연속될 때만 반영하세요(R7). 값이 너무 크면 반응이 느려지니 3~5에서 보정.
- **안전 정지** — 손이 사라지면 인식 노드가 `0`을 보내고 코디네이터가 정지 명령을 냅니다. 또한 `dc_motor_node`의 워치독으로 명령이 끊겨도 멈춥니다(R9). 통합 후 `Ctrl+C` 시 바퀴가 도는지 반드시 확인.
- **핀 충돌 주의** — DC(L293D)·스테퍼(ULN2003)·LCD(I2C)를 동시에 쓰면 핀이 겹칠 수 있습니다. Day 3 핀맵을 기준으로 **함께 구동할 장치끼리 핀이 충돌하지 않게** 조정하세요. 모터 전원은 **외부 전원 + GND 공통**.
- **좌우 반전** — 셀카 미러링이면 엄지 판정 부등호가 반대가 됩니다. `/gesture/image`로 확인 후 부호를 맞추세요.
- **도메인 분리** — 각자 `ROS_DOMAIN_ID`를 다르게 두어 다른 사람과 토픽이 섞이지 않게 하세요.

| 증상 | 원인 | 해결 |
|---|---|---|

| 엄지/좌우가 반대 | 셀카 미러링 | `cv2.flip(frame,1)` 확인, 엄지 부등호 반전(Day 4) |
| 스테퍼가 끝없이 돌아감 | 매 프레임 발행 | `changed` 분기 안에서만 `/stepper_steps` 발행 |
| 스테퍼가 안 멈춤/역방향 | 스텝 수·시퀀스 | `STEP_ONE_REV` 부호·값 확인(양수=정방향, Day 3) |
| DC 모터가 안 멈춤(Ctrl+C 후) | 워치독·토픽명 | `dc_motor_node` 워치독·`/cmd_vel` 이름 일치(Day 3) |
| LCD에 글자 안 나옴 | I2C 주소·대비 | `i2cdetect`로 0x27/0x3f 확인, 백팩 가변저항 조절(Day 3) |
| 카메라 미인식 | 장치·VM 패스스루 | `ls /dev/video*`, VirtualBox는 웹캠 attach(Day 4-0) |
| `mediapipe` import 오류 | 파이썬 버전 | Python 3.8~3.11, `pip3 install mediapipe`(Day 4) |
| Pi가 모터 구동 시 꺼짐 | 전류로 전압 강하 | 모터 외부 전원 분리, GND 공통(Day 3) |
